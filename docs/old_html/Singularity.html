Singularity is a Linux software that allows application-level isolation. You can build a container filled with whatever applications you wish outside of the duke cluster, then move the container to the cluster to run the programs.

 

Commands to be run outside of Singularity container

Commands to be run inside container

 

duke compute cluster is running singularity 2.3.2 as of 10/30/2017. (Duke cluster upgraded to 2.4 in 12/17. These build instructions should still work, but now you can run 2.4 containers on the server. See email below.)

New versions of singularity use build instead of create & bootstrap commands. Just amend some of the commands below, but the overall procedure will be the same.

 

install that version on a computer you have root access to (VMBox works)

 

sudo singularity create -s 12000 deb.simg (size in MB, you can expand the container, so start small and add more space as needed)

 

sudo singularity bootstrap deb.simg spec

 

where spec contains the recipe

 

Bootstrap:docker

From: debian:jessie

 

%post

    mkdir /dscrhome

    mkdir /global

    mkdir /matlab

 

/dscrhome is required for singularity on duke computer cluster

in host, download matlab zip from mathworks

 

mkdir global, and put zip there

 

sudo singularity shell --writable -B global:/global deb.simg

 

Note, X system should be working, you can test this by running xeyes, xcalc, or xclock.

If these aren't installed, install x11-apps

You should see these gui apps running. If not, try binding the home directory, sudo singularity shell --writable -B global:/global -B /home/:/home/ deb.simg

 

%% now inside container

apt-get update

apt-get install xorg unzip

 

mv /global/matlab.zip /matlab

 

cd /matlab

 

unzip matlab.zip

 

./install

 

Don't activate Matlab quite yet, unless you need to test the installation.

 

complete installation.

 

We must make licenses folder write enabled for all, for manual activation on the duke cluster

 

chmod -R 777 /usr/local/MATLAB/R2017b

chmod 777 /dscrhome

chmod 777 /global

 

exit container

 

sudo tar cjvf deb.simg.tar.bz2 deb.simg (slower, smaller file)

or

sudo tar czvf deb.simg.tar.gzip deb.simg (much faster, larger file)

 

move tar to pace system using browswer ftp,

pacedata.duhs.duke.edu/

 

log into duke cluster using ssh -X to enable X forwarding

 

make sure pace storage is mounted (see PACE) and you can see coganlab/ folder.

 

cd coganlab/

 

tar xvf deb.simg.tar.bz2 to uncompress into coganlab/ folder

 

singularity shell --writable deb.simg

 

cd /usr/local/MATLAB/R2017b/bin

 

./activation_matlab.sh, choose license.lic file.

 

If you don't have license file, you need to create one through mathworks.com. Click the help button

and it will give you information on how to obtain license through mathworks.com.

(includes hostname ID, which I believe is just mac address of network interface)

ip link show, look at link/ether string under eth0 device

e.g. 005056aeb9c8

 

Upload that license.lic file to pace. Then re-run activation_matlab.sh and point to that license file.

 

You may need to launch the shell with the --writable flag from now on, e.g.  singularity shell --writable deb.simg

 

At some point you might wish to downsize your container,

 

while in container, run df / to see how much space you are using (add some buffer space, maybe 2 gb)

create a new container with desired size

 

sudo singularity create -s 10000 deb2.simg

 

then pipe the content from old to new

 

sudo singularity export deb.simg | sudo singularity import deb2.simg

 

To uninstall singularity, run the following:

sudo rm -rf /usr/local/libexec/singularity

sudo rm -rf /usr/local/etc/singularity

sudo rm -rf /usr/local/include/singularity

sudo rm -rf /usr/local/lib/singularity

sudo rm -rf /usr/local/var/lib/singularity/

sudo rm /usr/local/bin/singularity

sudo rm /usr/local/bin/run-singularity

sudo rm /usr/local/etc/bash_completion.d/singularity

sudo rm /usr/local/man/man1/singularity.1

All:

 

You’re getting this email because at one time or another you used the CI process on gitlab.oit.duke.edu with the DCC CI Runner to build a Singularity container. This runner used version 2.3.1 of Singularity. At the end of August Docker changed the format of their Docker containers on Docker Hub. This broke Singularity version 2.3.1. Singularity released a version 2.3.2 (which is installed on most of the servers) that was able to use the new format. Unfortunately, I didn’t catch that the version on the runner stayed the same so any builds that used a container with the new format failed. Everyone must have already had their working containers built as there were no complaints till this week. We’ve updated the DCC GitLab CI Runner to version 2.4.1 of Singularity. I will be pushing that version out to the servers that have Singularity installed. There’s no need to rebuild your containers if they are working, version 2.4.1 will run containers built with an earlier version.

 

This updated version of Singularity has some new features (see http://singularity.lbl.gov/index.html for documentation). One new feature is that the containers now use Squashfs. This means you don’t have to prebuild an image that you install into the new build command will create the container with the required space. It also makes the container read only. If you need a writable container you can convert it. Also, Singularity has changed their recommended naming convention for the definition files from Singularity.def to just Singularity. With these new features in mind the CI build scripts were modified. Since Squashfs is already compressed there is no longer a need to compress the built image. If you use the new process you can just curl or wget the built image directly and then you won’t have to extract it from a “.tar.gz” anymore. If you have old projects that you don’t want to modify they will still work, it’s just the “.tar.gz” file will not be that different in size than the extracted container. The following is the recommended “.gitlab-ci.yml” to put in your projects:

 

 

stages:

  - build

  - deploy

 

build_image:

  stage: build

  script:

    - build_image

  tags:

    - dcc

  artifacts:

    paths:

      - $CI_PROJECT_NAME.img

    expire_in: 1 day

 

deploy_image:

  stage: deploy

  script:

    - deploy_output

  tags:

    - dcc
